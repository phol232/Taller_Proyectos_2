# Especificación de Requerimientos – Planner UC

## 1. Visión General del Proyecto

*Planner UC* es un sistema web (arquitectura SPA + API REST) con motor de satisfacción de restricciones (CSP) que genera automáticamente horarios académicos válidos para universidades con currículo flexible.

El sistema gestiona cuatro entidades base (estudiantes, docentes, cursos, aulas), valida restricciones académicas en tiempo real y produce asignaciones libres de solapamientos de docente, aula y estudiante, respetando prerrequisitos y límites de créditos.

*Límites del PMV:* hasta 50 estudiantes, 20 docentes, 30 cursos y 20 aulas.
*Roles del sistema:* Administrador, Coordinador Académico, Docente y Estudiante.
*Restricción tecnológica de negocio:* compatibilidad obligatoria con Chrome, Firefox y Edge (versiones actuales); infraestructura de cómputo moderada sin cloud especializado.

| RF-05 | Validar Condiciones Académicas del Estudiante | *Trigger:* El sistema evalúa si un estudiante puede acceder a un curso. *Lógica:* Para cada prerrequisito declarado en el curso, el sistema verifica que el código del curso prerrequisito exista en la lista de cursos aprobados del estudiante. *Resultado:* Acceso permitido si todos los prerrequisitos están aprobados; acceso rechazado con lista de prerrequisitos faltantes en caso contrario. | *Dado que* el curso "CALC-2" requiere "CALC-1" como prerrequisito, *cuando* un estudiante sin "CALC-1" aprobado intenta agregarlo a su horario, *entonces* el sistema rechaza la operación e indica explícitamente el prerrequisito faltante. |
| RF-06 | Gestionar Disponibilidad Docente | *Trigger:* El Docente o Administrador registra o actualiza la disponibilidad horaria semanal de un docente. *Lógica:* El sistema almacena la disponibilidad como conjunto de franjas horarias (día, hora inicio, hora fin) marcadas como disponible o no disponible. *Resultado:* La disponibilidad queda registrada y el motor CSP la usa como dominio del docente durante la generación. | *Dado que* un docente marca la franja "lunes 08:00–10:00" como no disponible, *cuando* el motor CSP genera el horario docente, *entonces* no se produce ninguna asignación para ese docente en esa franja. |
| RF-07 | Generar Asignaciones Horarias de Cursos | *Trigger:* El Coordinador ejecuta la generación automática del horario docente para el período académico. *Lógica:* El motor CSP asigna a cada curso una combinación válida de docente, aula y franja horaria que satisface todas las restricciones hard definidas (sin solapamientos de docente ni de aula, dentro de disponibilidades registradas). *Resultado:* Conjunto de asignaciones sin solapamientos producido en ≤ 30 segundos para el escenario base del PMV, o notificación de imposibilidad con indicación de conflictos. | *Dado que* el escenario base está cargado (≤ 50 est., ≤ 20 doc., ≤ 30 cursos, ≤ 20 aulas), *cuando* el Coordinador ejecuta la generación, *entonces* el sistema produce un horario docente con cero solapamientos de docente y aula en ≤ 30 segundos, o notifica al usuario indicando los recursos en conflicto que impiden la solución. |
| RF-08 | Validar Horario Docente | *Trigger:* El sistema verifica las asignaciones generadas automáticamente o ajustadas manualmente por el Coordinador. *Lógica:* Para cada par de asignaciones del mismo docente o del mismo aula, el sistema verifica que sus franjas horarias no se solapen. *Resultado:* El horario es marcado como válido si no hay solapamientos, o se listan los conflictos detectados con detalle del recurso y franja involucrados. | *Dado que* el Coordinador asigna manualmente dos cursos al mismo docente en la misma franja horaria, *cuando* intenta confirmar el horario, *entonces* el sistema rechaza la confirmación e indica el conflicto de solapamiento de docente, especificando los cursos y la franja en conflicto. |
| RF-09 | Gestionar Concurrencia en la Asignación de Recursos | *Trigger:* Dos usuarios intentan asignar simultáneamente el mismo recurso (docente, aula o franja) a distintos cursos. *Lógica:* El sistema aplica mecanismos de reserva temporal y control de concurrencia (bloqueo optimista o pesimista) para garantizar que un recurso no sea comprometido por dos asignaciones simultáneas. *Resultado:* Exactamente una de las asignaciones concurrentes es aceptada; la otra recibe un error de conflicto de recurso. | *Dado que* dos Coordinadores intentan asignar simultáneamente el aula "A-101" en "lunes 08:00–10:00" a dos cursos distintos, *cuando* ambas solicitudes se procesan concurrentemente, *entonces* exactamente una asignación es aceptada y la otra recibe mensaje de error indicando conflicto de recurso. |
| RF-10 | Construir y Ajustar Horario Docente | *Trigger:* El Coordinador decide modificar una asignación generada o construir el horario manualmente. *Lógica:* El sistema presenta las opciones compatibles (docentes disponibles, aulas disponibles, franjas disponibles) para cada curso. El Coordinador selecciona la combinación deseada y el sistema valida en tiempo real (≤ 1 segundo) que la selección no produce solapamientos. *Resultado:* La asignación modificada se guarda en estado borrador sin afectar el horario publicado. | *Dado que* el Coordinador modifica la franja del curso "MAT-101" y selecciona una franja que produce solapamiento con otro curso del mismo docente, *cuando* intenta guardar el cambio, *entonces* el sistema indica el conflicto en ≤ 1 segundo e impide guardar la asignación inválida. |
| RF-11 | Confirmar o Cancelar Horario Docente | *Trigger:* El Coordinador decide confirmar o cancelar el horario docente en estado borrador. *Lógica:* Al confirmar: el sistema verifica cero solapamientos en todas las asignaciones del período y cambia el estado a "confirmado", bloqueando modificaciones sin cancelación previa. Al cancelar: el sistema libera todos los recursos reservados y elimina el borrador. *Resultado:* El horario queda en estado "confirmado" (recursos asignados) o "cancelado" (recursos liberados). | *Dado que* existe un horario docente en estado borrador con cero solapamientos, *cuando* el Coordinador confirma el horario, *entonces* el estado cambia a "confirmado" y los recursos quedan asignados. *Dado que* existe un horario confirmado, *cuando* el Coordinador lo cancela, *entonces* todos los recursos quedan liberados y disponibles para nueva asignación. |
| RF-12 | Generar Horario del Estudiante | *Trigger:* El Estudiante o Coordinador solicita la generación automática de una propuesta de horario para un estudiante. *Lógica:* El sistema selecciona una combinación válida de secciones de los cursos elegidos, verificando prerrequisitos, límite de créditos, disponibilidad de vacantes y ausencia de solapamientos. *Resultado:* Propuesta de horario producida en ≤ 5 segundos bajo condiciones normales, o notificación de imposibilidad indicando el motivo. | *Dado que* el estudiante ha seleccionado cursos válidos (prerrequisitos aprobados, créditos dentro del límite), *cuando* solicita la generación automática de su horario, *entonces* el sistema produce en ≤ 5 segundos una propuesta sin solapamientos y con vacantes disponibles, o notifica que no existe combinación válida indicando el conflicto. |
| RF-13 | Construir y Ajustar Horario del Estudiante | *Trigger:* El Estudiante decide agregar un curso, retirar un curso o cambiar de sección en su horario. *Lógica:* El sistema verifica en tiempo real (≤ 1 segundo) que cada cambio cumple prerrequisitos, límite de créditos, disponibilidad de vacantes y ausencia de solapamientos. *Resultado:* El cambio se aplica si es válido, o se rechaza con indicación del motivo específico. | *Dado que* el horario del estudiante acumula 20 créditos y el límite es 22, *cuando* el estudiante intenta agregar un curso de 5 créditos, *entonces* el sistema rechaza la operación e indica que se excedería el límite de créditos, mostrando el total resultante (25) vs. el límite (22). |
| RF-14 | Validar Restricciones del Horario del Estudiante | *Trigger:* El Estudiante realiza cualquier modificación en la construcción de su horario. *Lógica:* Tras cada acción, el sistema evalúa en ≤ 1 segundo: (1) prerrequisitos del curso agregado, (2) suma total de créditos vs. límite configurado, (3) vacantes disponibles en la sección seleccionada, (4) solapamiento con secciones ya incluidas en el horario. *Resultado:* El sistema muestra el estado de validación actualizado inmediatamente tras cada acción. | *Dado que* el estudiante tiene "CALC-1" en su horario en la franja "lunes 08:00–10:00" y agrega "FÍSICA-1" con sección en la misma franja, *cuando* confirma la adición, *entonces* el sistema detecta el solapamiento en ≤ 1 segundo, rechaza la operación e indica la franja y los cursos en conflicto. |
| RF-15 | Notificar Conflictos en la Generación o Ajuste del Horario | *Trigger:* El sistema no puede completar la generación o acepta una modificación por restricciones no satisfechas. *Lógica:* El sistema identifica la restricción específica que impide la operación (créditos excedidos, prerrequisito faltante, vacante agotada, solapamiento) y construye un mensaje descriptivo con el tipo de conflicto, el recurso involucrado y la restricción violada. *Resultado:* El usuario recibe el mensaje de error específico en ≤ 1 segundo tras la acción. | *Dado que* el motor CSP no encuentra solución para el horario docente en el tiempo límite, *cuando* finaliza la ejecución, *entonces* el sistema notifica al Coordinador con el mensaje: "No se encontró solución. Conflictos identificados: [lista de recursos en conflicto]". Los mensajes de error nunca muestran únicamente un código genérico sin descripción. |
| RF-16 | Visualizar Horarios | *Trigger:* Un usuario autenticado solicita visualizar un horario (docente, estudiante o general del período). *Lógica:* El sistema recupera las asignaciones confirmadas y las presenta en formato de grilla semanal (días × franjas horarias) indicando para cada bloque: nombre del curso, sección, docente asignado y aula. *Resultado:* La grilla se carga y muestra en ≤ 3 segundos bajo condiciones normales. | *Dado que* existen asignaciones confirmadas para el período, *cuando* el Estudiante accede a "Mi horario", *entonces* el sistema muestra en ≤ 3 segundos la grilla semanal con todos sus cursos confirmados, indicando para cada bloque: nombre del curso, sección, docente y aula. |
| RF-17 | Exportar Horarios | *Trigger:* Un usuario autenticado solicita exportar un horario en PDF o Excel. *Lógica:* El sistema genera el archivo del formato solicitado con el contenido completo del horario visualizado (curso, sección, docente, aula, franja horaria por día). *Resultado:* El archivo se genera y descarga en ≤ 5 segundos bajo condiciones normales. El archivo contiene al menos: código del curso, nombre, sección, docente, aula y franjas horarias. | *Dado que* el Coordinador visualiza el horario general del período, *cuando* solicita exportar en PDF, *entonces* el sistema genera y descarga un archivo PDF con el horario completo en ≤ 5 segundos, con los campos mínimos definidos. |
| RF-18 | Gestionar Autenticación y Acceso | *Trigger:* Un usuario intenta iniciar sesión o acceder a una funcionalidad del sistema. *Lógica:* El sistema valida credenciales (usuario + contraseña con hash seguro), genera sesión o JWT con tiempo de expiración configurado (≤ 8 horas) y verifica el rol del usuario antes de ejecutar cada operación. *Resultado:* Acceso concedido a funcionalidades permitidas por el rol; HTTP 401 si no autenticado, HTTP 403 si el rol no tiene permisos. | *Dado que* un usuario con rol "Estudiante" está autenticado, *cuando* intenta acceder al endpoint de "Gestionar Docentes" (exclusivo de Administrador), *entonces* el sistema retorna HTTP 403 y no ejecuta la operación. Un JWT expirado retorna HTTP 401 en el siguiente request autenticado. |

## 3. Árbol de Calidad y RNF

| Categoría arc42 | Atributo de Calidad | Requerimiento Cuantificable (Métrica) | Justificación de Negocio (¿Qué pasa si no se cumple?) |

| *Usabilidad* | Facilidad de aprendizaje | Un usuario nuevo, sin capacitación previa, completa las tareas de registro de una entidad y visualización de su horario en ≤ 5 minutos en una prueba cronometrada. | Si el sistema no es intuitivo, coordinadores y estudiantes continuarán usando procesos manuales. |
| *Usabilidad* | Accesibilidad web | La interfaz web cumple las pautas WCAG 2.1 nivel AA en los componentes de navegación principal, formularios y visualización de horarios. Verificable mediante herramienta automatizada de accesibilidad. | El incumplimiento de WCAG 2.1 AA puede excluir a usuarios con discapacidades, reduciendo el universo de usuarios. |
| *Compatibilidad* | Compatibilidad de navegadores | El sistema funciona sin errores de renderizado, funcionalidad o JavaScript en las versiones actuales de Chrome, Firefox y Edge en Windows y macOS. | Si el sistema solo funciona en un navegador, una parte significativa de los usuarios no puede acceder. |
| *Seguridad* | Autenticación y gestión de sesiones | El sistema implementa JWT con expiración ≤ 8 horas o sesiones seguras con flags HttpOnly + Secure. Las contraseñas se almacenan con bcrypt (cost factor ≥ 10). JWT expirado retorna HTTP 401 en el siguiente request autenticado. | Sin autenticación correcta, cualquier usuario puede acceder a datos académicos de otros, comprometiendo la privacidad. |
| *Seguridad* | Control de acceso por roles | Cada endpoint de la API verifica el rol del usuario autenticado antes de ejecutar la operación. Accesos fuera de rol retornan HTTP 403. Verificable mediante suite de pruebas de autorización por rol. | Sin control de acceso granular, un Estudiante podría modificar datos de docentes o confirmar horarios, comprometiendo la integridad del sistema. |
| *Seguridad* | Protección de datos y vulnerabilidades | El sistema mitiga las vulnerabilidades del OWASP Top 10 aplicables al alcance del PMV: inyección SQL (queries parametrizados u ORM), XSS (escape de salida), CSRF (tokens de sincronización o SameSite cookies) y exposición de datos sensibles (HTTPS obligatorio en producción). | La explotación de estas vulnerabilidades compromete la confidencialidad de datos académicos y la integridad del sistema. |
| *Confiabilidad* | Concurrencia y consistencia | El sistema maneja correctamente accesos concurrentes durante la generación y asignación de horarios: cero asignaciones duplicadas del mismo recurso bajo carga concurrente de hasta 5 usuarios simultáneos. Verificable mediante prueba de carga concurrente documentada. | Sin control de concurrencia, dos coordinadores pueden asignar el mismo recurso a dos cursos distintos simultáneamente, produciendo inconsistencias en el horario. |
| *Confiabilidad* | Consistencia transaccional | Las operaciones de reserva, confirmación, liberación y asignación se ejecutan de forma transaccional (commit completo o rollback completo). Ninguna operación queda en estado intermedio ante un error. Verificable mediante prueba de interrupción forzada durante una transacción. | Una transacción parcial puede dejar el sistema en estado inconsistente (recurso reservado sin asignación registrada), requiriendo corrección manual. |
| *Mantenibilidad* | Cobertura de pruebas | Los módulos críticos (motor CSP, validaciones de restricciones y autenticación/autorización) tienen cobertura de pruebas unitarias e integración ≥ 70%. Verificable mediante reporte de herramienta de testing al cierre de cada sprint. | Cobertura < 70% en módulos críticos aumenta el riesgo de regresiones no detectadas al modificar el motor de generación o las reglas de validación. |

## 4. Notas de Refinamiento

El siguiente registro documenta cómo los conceptos vagos de la especificación preliminar fueron convertidos en métricas SMART verificables.

| Concepto Vago Original | Problema de Ambigüedad | Métrica SMART Resultante |

| "Interfaz comprensible y operable" | No define qué acción, quién la realiza ni en cuánto tiempo. Es completamente subjetivo. | Un usuario nuevo completa registro de entidad + visualización de horario en ≤ 5 minutos en prueba cronometrada. |
| "Rendimiento adecuado en la generación de horarios" | No especifica qué tipo de horario, qué escenario de datos ni cuánto tiempo es aceptable. | Horario docente: ≤ 30 s. Horario estudiante: ≤ 5 s. Escenario base definido: ≤ 50 est., ≤ 20 doc., ≤ 30 cursos, ≤ 20 aulas. |

| "Cumplir WCAG 2.1 nivel AA" | No especifica qué componentes deben cumplirlo ni cómo verificarlo objetivamente. | Componentes de navegación principal, formularios y visualización de horarios deben pasar validación automatizada de WCAG 2.1 AA. |
| "Condiciones normales" (usado en múltiples RNF) | "Normal" no está definido; impide reproducir la prueba. | Condiciones normales = hasta 10 usuarios concurrentes para operaciones generales y hasta 5 usuarios concurrentes para operaciones de asignación. |
